# Bug Fixer — Автоматическое исправление регресс-багов

Читает `docs/testing/regression-cases/auto-result/` → находит последний `SUMMARY.md` → извлекает FAIL-баги → проверяет воспроизведение → фиксит → обновляет статус.

## Когда использовать

- После регресс-прогона — «исправь баги», «fix bugs from regression»
- После ответа разработчика о приоритетах — «fix high priority bugs only»
- Для конкретного бага — «fix --context-window 32000 fatal»
- CI/CD пайплайн — часть пост-регрессионного процесса

## Workflow

### 1. Найти последний SUMMARY.md

```bash
ls -dt docs/testing/regression-cases/auto-result/*/ | head -1
```

Прочитать `docs/testing/regression-cases/auto-result/<latest>/SUMMARY.md`.

Если SUMMARY.md нет → сообщить «Нет регресс-результатов. Сначала запусти regression-runner.»

### 2. Извлечь FAIL-баги

Из SUMMARY.md найти секцию `## Найденные баги (FAIL)`.

Для каждого бага извлечь:
- Название
- Файл и номер кейса
- Симптом
- Приоритет
- Команда для воспроизведения

### 3. Для каждого бага — цикл

#### Шаг A: Воспроизвести

```bash
<команда из бага>
```

- Если команда не падает / работает корректно → исправлено. Перейти к Шагу E.
- Если падает → перейти к Шагу B.

#### Шаг B: Анализ

Найти корневую причину:

1. **Если ошибка в выводе/сообщении** → grep по `src/` для поиска строки
2. **Если fatal/crash** → посмотреть стектрейс, найти файл и строку
3. **Кросс-валидация**: проверить unit-тесты смежных модулей
4. **Git blame**: `git log --oneline -5 -- <file>` — был ли недавний коммит, который мог внести баг

> **💡 Совет:** Перед правкой функции/класса запусти `ts-morph-analyzer` чтобы понять blast radius — кто вызывает этот код и какие модули зависят:
> ```bash
> bun run .soba/skills/ts-morph-analyzer/scripts/impact-analysis.ts <имя функции>
> bun run .soba/skills/ts-morph-analyzer/scripts/find-references.ts <имя класса>
> ```
> Результаты в `.soba/skills/ts-morph-analyzer/output/` — JSON с файлами, строками, контекстом.

```bash
# Поиск по симптомам
grep -rn "ошибка|стек|фраза" src/

# Если нужен стектрейс — посмотреть на вывод ошибки, найти файл
# Если ошибка валидации — найти код валидации
```

#### Шаг C: Исправить

Исправить код в `src/`:

| Тип бага | Подход |
|----------|--------|
| **Validation** (неправильные границы, hardLimit) | Исправить `validateCompactionConfig` или границы в конфиге |
| **Missing error handling** (--foobar, невалидный флаг) | Добавить валидацию в парсер аргументов |
| **ANSI output** (--no-color оставляет ANSI) | Найти где печатается ANSI даже с `noColor: true` и добавить условие |
| **Fatal вместо graceful** | Заменить `process.exit` или `throw` на graceful error + сообщение |
| **Logic error** | Исправить условие, алгоритм, перепроверить инвариант |

> **Правило:** Каждый фикс должен быть минимальным и точечным. Не рефакторить соседний код.

#### Шаг D: Подтвердить

```bash
# Unit-тесты затронутых модулей
bun test tests/<module>/<file>.test.ts

# Полный тест
bun test

# Воспроизведение бага
<команда из бага>

# Линтер
bun run lint
```

- Unit-тесты → PASS
- Команда из бага → больше не падает, поведение корректно
- `bun run build` → OK

Если не исправлено → вернуться к Шагу B. Если не получается исправить → отметить как ❌ Не исправлен и запросить помощь разработчика.

#### Шаг E: Обновить статус в SUMMARY.md

- Если баг исправлен: в разделе `## Статус исправлений` заменить `❌ Не исправлен` на `✅ Исправлен`
- В разделе `## Найденные баги` добавить строку `**Исправлено:** <что сделано, какой файл изменён>`
- В секции с ручным прогоном (FAIL таблица) заменить `☐ баг жив` на `✅ починен`

### 4. После всех багов

Показать девелоперу:

```markdown
## Результат исправления багов

| Баг | Статус | Изменения |
|-----|--------|-----------|
| --no-color | ✅ Исправлен | src/cli/args.ts: ... |
| --context-window 32000 | ❌ Не исправлен | Требуется рефакторинг валидации |
| --foobar | ✅ Исправлен | src/cli/args.ts: добавлена валидация |

### Вывод

<2 из 3 багов исправлены. Оставшийся требует перепроектирования валидации compaction config.>
```

Сохранить SUMMARY.md в том же файле (перезаписать).

### 5. Коммит (если есть исправления)

```bash
git add -A
git commit -m "fix(bug-fixer): <краткое описание исправлений>"
```

## Примеры

### Баг #1: `--no-color` оставляет ANSI

```bash
# Воспроизведение
bun run dist/cli.js --no-color "Say ok" | cat -v | head -3

# Вывод: ^[[1mWelcome to^[[0m — ANSI есть
# Анализ: grep -rn "bold\|dim\|ansi\|chalk\|fg\|bg\|style" src/ | grep -i color
# Находим: src/cli/args.ts -> noColor проверяется только в TUI, но не в форматерах
# Фикс: добавить проверку noColor в renderStatusBar / output functions
```

### Баг #2: `--context-window 32000` → fatal

```bash
# Воспроизведение
bun run dist/cli.js --context-window 32000 --no-session "Say ok"

# Fatal: Invalid compaction config: keepRecentTokens must be < hardLimit
# Анализ: grep -rn "keepRecentTokens.*hardLimit\|Invalid compaction" src/
# Находим: src/core/config/config-loader.ts строка 213 -> validateCompactionConfig
# Причина: keepRecentTokens = contextWindow * 0.5, hardLimit = contextWindow
# При contextWindow=32000: keepRecentTokens=16000, hardLimit=contextWindow=32000
# Проверка: keepRecentTokens(16000) < hardLimit(32000)? Должно быть true!
# Значит баг в другом месте или другая формула.
```

### Баг #3: `--foobar` молча игнорируется

```bash
# Воспроизведение
bun run dist/cli.js --foobar

# Вывод: Welcome to SOBA — без ошибки
# Анализ: читаем src/cli/args.ts -> parseArgs
# Находим: неизвестные флаги молча игнорируются (нет ветки else)
# Фикс: добавить проверку на неизвестный флаг -> console.error + process.exit(1)
```

## Правила

- **Не исправлять то, что не сломалось** — только FAIL-баги из SUMMARY.md
- **Минимальные изменения** — не рефакторить соседний код
- **После каждого фикса — тесты** — `bun test` + `bun run build` + проверка воспроизведения
- **Если баг не воспроизводится** — пометить ✅ Исправлен с пометкой «не воспроизводится»
- **Если баг не фиксится** — пометить ❌ Не исправлен с объяснением причины
- **Не менять регресс-кейсы** — только код и SUMMARY.md
