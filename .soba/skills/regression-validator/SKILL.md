# Regression Validator — Валидация регресс-кейсов перед прогоном

Проходит по файлам регресс-кейсов в `docs/testing/regression-cases/` и проверяет их на соответствие текущей реализации. Выявляет расхождения: несуществующие тестовые файлы, удалённые CLI-флаги, переименованные функции, неактуальные кейсы. Запускается **перед** regression-runner, чтобы гарантировать что кейсы не расходятся с кодом.

> **Dev-only** — этот скилл только для разработки SOBA Agent. Не попадает в bundled скилы продукта.

## Когда использовать

- **Перед регресс-прогоном** — «свалидируй регресс», «validate regression cases»
- **После рефакторинга** — проверить что кейсы не разошлись с кодом
- **После добавления/удаления фич** — найти устаревшие или отсутствующие кейсы
- **Периодически** — аудит качества регресс-кейсов
- **Для конкретного файла** — «проверь кейсы 13»

## Workflow

### 1. Определить область проверки

```bash
# Список всех файлов кейсов
ls docs/testing/regression-cases/*.md | grep -v INDEX.md | grep -v auto-result

# Если указан конкретный файл — только он
# Например: docs/testing/regression-cases/13-sessions.md
```

- **По умолчанию**: все 36 файлов по порядку (01–36)
- **Если указан номер**: `"проверь кейсы 13"` → только `13-sessions.md`
- **Если указан диапазон**: `"проверь кейсы 01-05"` → файлы 01–05

### 2. Пошаговая валидация (для каждого файла)

> **Правило:** один файл → один шаг. Не читать все файлы заранее.

Для **каждого** файла из выбранной области выполнять цикл:

#### Шаг A: Прочитать файл кейса

Прочитать `docs/testing/regression-cases/<file>.md`.

Извлечь:
- Заголовок, номера кейсов
- Все упоминания тестовых файлов (`tests/...`)
- Все упоминания CLI-флагов (`--flag`)
- Все упоминания инструментов (`ls`, `read`, `write`, `edit`, `bash`, `checkpoint`, `activate_skill`)
- Все упоминания slash-команд (`/help`, `/clear`, `/session`, ...)
- Все упоминания путей к исходникам (`src/...`)
- Все упоминания конфигурационных ключей
- Номер версии (если указан)

#### Шаг B: Выполнить проверки

Для каждого извлечённого элемента выполнить соответствующую проверку:

##### B1. Проверка тестовых файлов

Если кейс ссылается на тестовый файл (явно или по контексту — например, кейс про сессии → `tests/core/session/`):

```bash
# Проверить существование
ls tests/<module>/<file>.test.ts 2>/dev/null && echo "EXISTS" || echo "MISSING"

# Если файла нет — поискать похожие
find tests/ -name "*<keyword>*" -type f 2>/dev/null
```

**Вердикты:**
- Файл существует → ✅ **VALID**
- Файл переименован/перемещён → ⚠️ **MISMATCH** (указать актуальный путь)
- Файл отсутствует → ❌ **STALE** (тесты удалены, кейс устарел)

##### B2. Проверка CLI-флагов

Если кейс упоминает CLI-флаг (`--model`, `--session`, `--continue`, ...):

```bash
# Проверить что флаг определён в коде
grep -rn '"<flag>"' src/cli/ 2>/dev/null
# или
grep -rn "'<flag>'" src/cli/ 2>/dev/null
# или
grep -rn '<flag>' src/cli/ 2>/dev/null
```

Также проверить через `--help`:

```bash
bun run dist/cli.js --help 2>&1 | grep -o '\-\-[a-z-]*' | sort -u
```

**Вердикты:**
- Флаг есть в `--help` и в коде → ✅ **VALID**
- Флаг есть в коде, но отсутствует в `--help` → ⚠️ **MISMATCH** (не задокументирован)
- Флага нет нигде → ❌ **STALE** (флаг удалён, кейс устарел)

##### B3. Проверка инструментов агента

Если кейс проверяет конкретный инструмент (`ls`, `read`, `write`, `edit`, `bash`, `checkpoint`, `activate_skill`):

```bash
# Проверить регистрацию инструмента в коде
grep -rn "name.*<tool>" src/core/tools/ 2>/dev/null
# или в едином реестре
grep -rn "'<tool>'" src/core/ 2>/dev/null
```

**Вердикты:**
- Инструмент зарегистрирован → ✅ **VALID**
- Инструмент не найден → ❌ **STALE** (инструмент удалён)

##### B4. Проверка slash-команд

Если кейс проверяет slash-команду (`/help`, `/clear`, `/compact`, `/rewind`, `/capsule`, `/skill`, `/budget`, `/session`, `/exit`, `/theme`, `/lang`, `/project-trust`, `/queue`, `/permissions`, ...):

```bash
# Проверить регистрацию команды
grep -rn "'/<command>'" src/tui/ 2>/dev/null
grep -rn "\"/<command>\"" src/tui/ 2>/dev/null
```

И свериться с таблицей TUI-покрытия в regression-runner SKILL.md.

**Вердикты:**
- Команда зарегистрирована → ✅ **VALID**
- Команда не найдена → ❌ **STALE** (команда удалена)
- Команда есть, но нет теста → ⚠️ **MISMATCH** (не покрыта тестами)

##### B5. Проверка путей к исходникам

Если кейс упоминает путь к исходному файлу (`src/core/session/session-manager.ts`, `src/config/...`):

```bash
# Проверить существование
ls <путь> 2>/dev/null && echo "EXISTS" || echo "MISSING"

# Если нет — поискать похожие
find src/ -name "*<keyword>*" -type f 2>/dev/null
```

**Вердикты:**
- Файл существует → ✅ **VALID**
- Файл переименован/перемещён → ⚠️ **MISMATCH** (указать актуальный путь)
- Файл отсутствует → ❌ **STALE**

##### B6. Проверка версии

Если кейс содержит конкретную версию (например, `v0.3.2`):

```bash
# Текущая версия
bun run dist/cli.js --version 2>/dev/null || node -e "console.log(require('./package.json').version)"
```

**Вердикты:**
- Версия совпадает → ✅ **VALID**
- Версия не совпадает → ⚠️ **MISMATCH** (указать актуальную версию)

##### B7. Проверка конфигурационных ключей

Если кейс ссылается на ключи конфигурации (`apiKey`, `baseUrl`, `model`, ...):

```bash
# Найти где определяется схема конфигурации
grep -rn "<key>" src/config/ 2>/dev/null
```

**Вердикты:**
- Ключ существует → ✅ **VALID**
- Ключ не найден → ❌ **STALE**

#### Шаг C: Классифицировать результат

После всех проверок определить общий вердикт для файла:

| Статус | Описание |
|--------|----------|
| **🟢 VALID** | Все проверки прошли, расхождений нет |
| **🟡 MINOR** | Есть незначительные расхождения (версия, путь перемещён) — кейс актуален, но нужно обновить пару строк |
| **🟠 STALE** | Есть устаревшие ссылки (удалённый флаг, инструмент, команда) — кейс требует пересмотра |
| **🔴 BROKEN** | Большинство ссылок битые, файл не соответствует реализации — кейс нужно переписать |

#### Шаг D: Сохранить результат

Записать в `docs/testing/regression-cases/auto-result/validation/YYYY-MM-DD/<file>.md`:

```markdown
# Валидация регресс-кейсов: <Название>

**Файл:** <имя файла>
**Дата:** YYYY-MM-DD
**Общий вердикт:** 🟡 MINOR
```

> Если файл полностью **VALID** — можно пропустить детальные результаты (только сводка и вердикт).

#### Шаг E: Перейти к следующему файлу

Повторить шаги A–D для следующего файла.

### 3. Поиск пробелов в покрытии (после валидации всех файлов)

После проверки всех кейсов — поискать функции без регресс-кейсов:

```bash
# Найти все модули в src/
find src/ -name "*.ts" -not -name "*.test.ts" -not -name "*.d.ts" | sort > /tmp/src-modules.txt

# Извлечь все тестовые файлы, упомянутые в кейсах
grep -roh 'tests/[a-z0-9_/.-]*\.test\.ts' docs/testing/regression-cases/*.md | sort -u > /tmp/tests-in-cases.txt

# Найти тесты, не упомянутые в кейсах
find tests/ -name "*.test.ts" | sort > /tmp/all-tests.txt
comm -13 /tmp/tests-in-cases.txt /tmp/all-tests.txt
```

Также проверить крупные фичи по `--help` и `src/`:

```bash
# Посмотреть список slash-команд в --help или коде
grep -rn '"/' src/tui/ | grep -o '"/[a-z-]*"' | sort -u
```

**Результат** — список потенциально непокрытых областей:

```markdown
## Пробелы в покрытии

### Тесты без регресс-кейсов
- `tests/core/new-feature.test.ts` — нет кейса
- `tests/tui/new-command.test.ts` — нет кейса

### Модули без регресс-кейсов
- `src/core/new-module.ts` — не упоминается в кейсах
```

## Правила валидации

### Что проверяется

| Категория | Что ищем в кейсе | Где проверяем |
|-----------|-----------------|---------------|
| **Тестовые файлы** | `tests/...` | `ls` / `find tests/` |
| **CLI-флаги** | `--flag` | `--help` вывод + `grep src/cli/` |
| **Инструменты** | `ls`, `read`, `write`, `edit`, `bash`, `checkpoint`, `activate_skill` | `grep src/core/tools/` |
| **Slash-команды** | `/command` | `grep src/tui/` |
| **Исходники** | `src/...` | `ls` / `find src/` |
| **Версия** | `vX.Y.Z` | `--version` / `package.json` |
| **Конфиг-ключи** | `apiKey`, `baseUrl`, `model`, ... | `grep src/config/` |

### Что НЕ проверяется

- **Логика тестов** (проходят ли тесты) — это задача regression-runner
- **Корректность поведения** — это задача regression-runner
- **API-ответы** — это задача regression-runner
- **Рендеринг TUI** — это задача regression-runner
- **Форматирование кейсов** (грамматика, стиль) — не проверяется

### Приоритет расхождений

| Тип расхождения | Действие |
|----------------|----------|
| **STALE** (удалённый флаг/инструмент) | 🔴 Блокирует regression-runner — кейс нужно исправить |
| **MISMATCH** (переименован/перемещён) | 🟡 Можно запускать regression-runner, но с note |
| **Версия устарела** | 🟡 Можно запускать, но обновить |
| **Пробел в покрытии** | ℹ️ Информационно, не блокирует |

## Интеграция с другими скиллами

| Скилл | Как взаимодействует |
|-------|-------------------|
| **regression-runner** | Запускается ПОСЛЕ validator. Если validator нашёл STALE/BROKEN — runner не запускается. |
| **bug-fixer** | Использует результаты validator для понимания контекста багов |
| **ts-morph-analyzer** | Можно использовать для поиска references при валидации исходников |
| **ts-error-janitor** | После удаления мёртвого кода — перевалидировать кейсы |

## Пример использования

Разработчик: «Свалидируй регресс»

5. Показать итоговую сводку и готовность к regression-runner
