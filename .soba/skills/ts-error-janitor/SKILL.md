# TypeScript Error Janitor — Массовый фикс накопившихся TS-ошибок

Проходит по кодовой базе, группирует ошибки `tsc --noEmit` по типу и файлам, исправляет оптимальными порциями, отслеживает прогресс.

> **Dev-only** — этот скилл только для разработки SOBA Agent. Не попадает в bundled скилы продукта.

## Расположение

- **Скилл**: `.soba/skills/ts-error-janitor/SKILL.md`
- **Прогресс**: `.soba/skills/ts-error-janitor/progress.json`
- **Лог ошибок**: `docs/testing/ts-errors/`

## Быстрый старт

Скажи агенту:

```
Почини TS ошибки
```

или

```
Run TS janitor
```

Агент автоматически:
1. Запустит `bunx tsc --noEmit` и соберёт ошибки
2. Сгруппирует по типу и приоритету
3. Возьмёт первую порцию (каскадные типы → импорты → остальное)
4. Исправит файлы и сохранит прогресс
5. Перезапустит `tsc` для проверки
6. Повторит, пока ошибки не закончатся или не останутся только сложные кейсы

## Алгоритм: почему не «всё разом»

Если скормить модели 200 ошибок из 40 файлов — она:
- Потеряет контекст на середине
- Исправит симптомы, а не причину
- Создаст новые ошибки быстрее, чем исправит старые

**Правильный подход — водопад с обратной связью:**

```
tsc --noEmit → группировка → порция N файлов → фикс → tsc --noEmit → следующая порция
```

## Группировка ошибок (приоритет фикса)

| Приоритет | Тип ошибки | Почему первым | Пример |
|-----------|-----------|---------------|--------|
| **P0** | `TS2304: Cannot find name` (missing types) | Одна дыра в `.d.ts` порождает 50 ошибок внизу | Забытый `declare module`, отсутствующий `types` в tsconfig |
| **P1** | `TS2307: Cannot find module` | Каскадные импорты, ломают целые деревья файлов | Неправильный `alias`, отсутствующий `index.ts` |
| **P2** | `TS2322: Type 'X' is not assignable` | Часто следствие P0/P1, но иногда реальная логика | Несовпадение интерфейсов после рефакторинга |
| **P3** | `TS2345: Argument of type...` | Локальные ошибки в функциях | Поменялась сигнатура, забыли обновить вызов |
| **P4** | `TS7006: Parameter implicitly has an 'any'` | Strict-mode проверки, не ломают компиляцию | Добавить типы в callback'и |
| **P5** | `TS6133: 'X' is declared but never used` | Чистка, можно оставить на потом | Неиспользуемые импорты/переменные |

## Размер порции

- **P0–P1:** до 3 файлов за вызов (каскадные, нужен глубокий анализ)
- **P2–P3:** до 5 файлов за вызов
- **P4–P5:** до 10 файлов за вызов (механические правки)

После каждой порции — **обязательный** `bunx tsc --noEmit`. Если ошибок стало больше — откатить и уменьшить порцию.

## Команды

### Полный прогон
```
bunx tsc --noEmit 2>&1 | tee docs/testing/ts-errors/tsc-output-latest.txt
```

### Подсчёт по типу
```bash
bunx tsc --noEmit 2>&1 | grep -oE 'TS[0-9]{4}' | sort | uniq -c | sort -rn
```

### Подсчёт по файлу
```bash
bunx tsc --noEmit 2>&1 | grep -oE '\([^)]+\.ts:[0-9]+:[0-9]+\)' | sed 's/:.*//' | sort | uniq -c | sort -rn
```

## Формат работы агента

### Шаг 1: Разведка
Агент запускает:
```bash
bunx tsc --noEmit 2>&1 | tee docs/testing/ts-errors/tsc-output-$(date +%Y%m%d-%H%M%S).txt
```

Анализирует:
- Сколько всего ошибок
- Какие коды доминируют
- Какие файлы — «хабы» (импортируются всюду)

### Шаг 2: Выбор порции
Берёт 1–3 файла с P0/P1 ошибками, которые импортируются из 5+ других файлов. Исправляет **причину**, а не симптом.

> **💡 Совет:** Чтобы найти файлы-хабы (импортируются из многих модулей), используй `ts-morph-analyzer`:
> ```bash
> bun run .soba/skills/ts-morph-analyzer/scripts/dependency-graph.ts
> ```
> Результат в `.soba/skills/ts-morph-analyzer/output/dependency-graph.json` — сортировка по `importedByCount`. Начинай с топ-хабов — исправление одного hub-файла может починить десятки каскадных ошибок.
> 
> Для чистки P5 (неиспользуемый код) — запусти dead-code анализ перед удалением:
> ```bash
> bun run .soba/skills/ts-morph-analyzer/scripts/dead-code.ts
> ```

### Шаг 3: Фикс
Для каждого файла:
1. Прочитать файл
2. Найти ошибку в контексте (не по номеру строки — они сдвинутся)
3. Применить минимальный фикс
4. Записать обратно

### Шаг 4: Верификация
```bash
bunx tsc --noEmit
```
- Если ошибок уменьшилось — сохранить прогресс, взять следующую порцию
- Если увеличилось — `git checkout -- <files>` и уменьшить порцию вдвое

### Шаг 5: Runtime check (после каждой фазы)
```bash
bun test
```
TS-фикс не должен ломать runtime. Если тесты падают — приоритет: откат TS-фикса, потом разбираться.

## Когда остановиться

| Условие | Действие |
|---------|----------|
| Ошибок 0 | Закончить. Добавить `bunx tsc --noEmit` в pre-commit hook. |
| Остались только P4–P5 | Можно остановиться и дать `--skipLibCheck` или `@ts-ignore` с TODO |
| Остались файлы из `skipped` | Не пытаться фиксить автоматически. Создать тикеты. |
| Ошибок не уменьшается 3 цикла подряд | Остановиться. Вероятно, требуется рефакторинг, не типофикс. |

## Примеры фиксов

| Ошибка | Причина | Фикс |
|--------|---------|------|
| `TS2304: Cannot find name 'Bun'` | Нет `@types/bun` | `bun add -d @types/bun` + `tsconfig.types` |
| `TS2307: Cannot find module '@soba/core'` | Alias не резолвится | Проверить `tsconfig.json` → `paths` |
| `TS2322: Type 'string' is not assignable to type 'ModelName'` | Расширяли enum, забыли union | Добавить в union или использовать `as const` |
| `TS2345: Argument of type 'X' is not assignable to parameter of type 'Y'` | Поменялась сигнатура функции | Обновить вызов или добавить overload |
| `TS7006: Parameter 'ctx' implicitly has an 'any' type` | Strict mode | Добавить тип: `ctx: AgentContext` |

## Безопасность

- **Не удалять файлы.** Только править типы.
- **Не менять логику.** Если фикс требует `if/else` или изменения алгоритма — внести в `skipped` и создать тикет.
- **Не использовать `@ts-ignore` массово.** Максимум 1–2 на весь проект для legacy-зависимостей.
- **Всегда `bun test` после фазы.** Типы ≠ работающий код.

## После завершения

1. Добавить в `package.json`:
```json
"scripts": {
  "typecheck": "tsc --noEmit"
}
```
2. Добавить в CI / pre-commit:
```bash
bun run typecheck
```
3. Удалить `docs/testing/ts-errors/` (архивные логи) или перенести в `.gitignore`.
