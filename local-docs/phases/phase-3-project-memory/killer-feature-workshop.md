# Killer Feature Workshop — что реально выделит SOBA

**Дата:** июнь 2026
**Контекст:** Proactive compaction и adaptive skills — «under the hood» фичи. Пользователь их не видит при первом запуске. Нужен **visible killer feature**, ради которого человек скажет «вау» в первые 30 секунд.

---

## 1. Почему компакция и скилы — НЕ киллер-фичи

| Фича | Проблема |
|------|----------|
| **Proactive compaction** | Решает проблему длинных сессий. Но 80% пользователей _не знают_, что это проблема — они привыкли рестартовать сессию каждые 2 часа. Фича невидима до 3-го часа работы. |
| **Context Capsules** | Технически круто, но для пользователя — абстракция. «Что такое капсула?» — уже вопрос. |
| **Adaptive Skills** | Нишевая фича для техлидов. 90% разработчиков не пишут скилы — они используют MCP или просто промптят. |
| **BYOK / любая модель** | Уже есть у OpenCode (75+ провайдеров), Cline, Aider. Не дифференциатор. |

**Вывод:** У SOBA есть архитектурные преимущества, но нет **visible killer feature** — того, что пользователь видит в первые 30 секунд и говорит «вау, хочу это».

## 3. Кандидаты в Killer Feature для SOBA

### Кандидат A: «Fix Until Green» — Автономный отладчик

**Концепция:** SOBA не просто пишет код — она **добивается, чтобы он работал**.

**Как это выглядит:**
1. Пользователь: «Напиши функцию parseCSV»
2. SOBA пишет код → запускает → находит ошибку → чинит → перезапускает → тесты проходят → **показывает зелёную галочку**
3. Пользователь видит не просто код, а **работающий код**

**Что нужно:**
- Интеграция с рантаймом проекта (bun test, npm test, pytest)
- Чтение stderr/stdout и автоматический фикс
- Цикл: generate → run → detect error → fix → rerun → ... → pass
- Лимит итераций с fallback

**Дифференциация:** Никто не делает это системно. Claude Code пробует, но не гарантирует. Aider пишет и коммитит, но не проверяет.

**Риски:** Может зациклиться; дорого по токенам; сложно для больших изменений.

---

### Кандидат B: «Codebase Aware» — Интеллект всего проекта

**Концепция:** SOBA знает весь код проекта, а не только то, что поместилось в контекст.

**Как это выглядит:**
1. Пользователь: «Где у нас используется PaymentService?»
2. SOBA: «PaymentService используется в 14 файлах: checkout.ts (строчка 23), billing/service.ts (строка 89)… Основная логика в src/payments/service.ts. Хочешь, покажу архитектуру?»
3. Пользователь: «Добавь логирование во все вызовы»
4. SOBA находит все 14 мест, правит, проверяет тесты

**Что нужно:**
- Индексация кодовой базы (сбор imports/exports, структура, типы)
- RAG или embedding-based retrieval для поиска релевантного кода
- Умная навигация через read с автоопределением границ

**Дифференциация:** 
- Cursor делает codebase indexing, но только внутри своей IDE
- Copilot делает codebase search, но не агентно
- OpenCode/Claude Code полагаются только на контекст

**Риски:** Сложность индексирования (языки, фреймворки, монорепы); затраты на RAG.

---

### Кандидат C: «Multi-Agent Team» — Твой dev team в терминале

**Концепция:** Не один агент, а команда специализированных агентов, работающих параллельно.

**Как это выглядит:**
1. Пользователь: «Добавь REST API эндпоинт для users и напиши тесты»
2. **Planner** разбивает задачу
3. **Coder** пишет эндпоинт
4. **Tester** параллельно пишет тесты
5. **Reviewer** проверяет код и тесты
6. **Runner** запускает тесты, фиксит баги
7. Пользователь получает готовый PR

**Что нужно:**
- Agent orchestrator — координация множества агентов
- Shared context между агентами
- Конфликт-резолюция (когда кодер и тестер меняют один файл)

**Дифференциация:**
- Devin делает нечто похожее, но enterprise-only и чёрный ящик
- Antigravity делает parallel agents, но только в GUI
- Ни один open-source CLI агент не делает multi-agent

**Риски:** Сложность координации; дороговизна (×3-4 токенов); UX — как показывать параллельную работу?

---

### Кандидат D: Visual-First Frontend Agent (оригинальный Phase 3)

**Концепция:** SOBA видит UI, который производит, и принимает решения на основе визуального анализа.

**Как это выглядит:**
1. Пользователь: «Сделай страницу логина по этому макету»
2. SOBA пишет код → запускает dev server → делает скриншот → сравнивает с макетом → правит CSS → повторяет
3. Показывает: «Готово. Отличия от макета: цвет кнопки #2563eb вместо #3b82f6, отступ 24px вместо 16px»

**Что нужно:**
- Headless browser (Chromium via Playwright/Puppeteer)
- Screenshot-as-tool
- Visual diff / pixel comparison
- Baseline management

**Дифференциация:** 
- Ни один CLI-агент этого не делает
- Только IDE-агенты (Cursor/Windsurf) могут показать результат, но не сравнить с макетом

**Риски:** Сложность infra (нужен Chromium); только для фронтенда.

---

### Кандидат E: Hybrid — «Ship-Ready Agent»

**Концепция:** Комбинация A + B — агент, который понимает _весь проект_ и _доводит код до рабочего состояния_.

**Как это выглядит:**
1. Пользователь: «Добавь сортировку по дате в таблицу заказов»
2. SOBA находит компонент таблицы (через codebase index)
3. Добавляет сортировку
4. Запускает тесты
5. Находит 2 упавших теста
6. Чинит их
7. Запускает билд
8. Всё зелёное

**Дифференциация:** 
- Сombination не делает НИКТО
- «Напиши — и оно работает» — Святой Грааль агентов

**Риски:** Сложность реализации обеих фич.

## 4. Рекомендация для Phase 3

### Что делать прямо сейчас (Phase 3 — TUI Polish)

TUI Polish остаётся правильным шагом — без хорошего UX никто не увидит killer feature. Но **добавить в design план**:

1. **ProviderRegistry** — базово, без этого не работает BYOK
2. **Fix-Until-Green (MVP)** — минимальная версия: запуск тестов после изменений, автофикс ошибок, отчёт
3. **TUI для отладки** — показать прогресс fix-until-green (статусы: запускаю → упало → чинят → зелёный)

### Почему Fix-Until-Green (Candidate A) — лучший выбор для Phase 3

| Критерий | A: Fix-Until-Green | B: Codebase Aware | C: Multi-Agent | D: Visual Preview | E: Hybrid (A+B) |
|----------|-------------------|-------------------|----------------|-------------------|------------------|
| **WOW-фактор** | 🟢 Высокий | 🟡 Средний | 🟢 Высокий | 🟡 Средний | 🟢 Очень высокий |
| **Сложность** | 🟡 Средняя | 🔴 Высокая | 🔴 Очень высокая | 🔴 Высокая | 🔴 Очень высокая |
| **Длина до результата** | 🟢 2-3 задачи | 🔴 5+ задач | 🔴 8+ задач | 🔴 5+ задач | 🔴 8+ задач |
| **Уникальность** | 🟡 Средняя (Claude Code пробует) | 🟡 Средняя (Cursor делает) | 🟢 Высокая | 🟢 Высокая | 🟢 Очень высокая |
| **Применимость** | 🟢 Весь код | 🟢 Весь код | 🟡 Только сложные задачи | 🔴 Только фронтенд | 🟢 Весь код |

**Итог:** **Fix-Until-Green** — самый быстрый путь к visible killer feature за минимальное время.

## 5. Что будет в промпте: «SOBA доводит код до зелёного»

Это тот месседж, который сразу понятен:

> **Другие агенты пишут код. SOBA делает так, чтобы он работал.**
> 
> Напиши функцию → SOBA запускает тесты → находит ошибку → чинит → перезапускает → 
> пока зелёная галочка. Без твоего участия.

---

## 6. Риски Fix-Until-Green

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Бесконечный цикл | Средняя | Max итераций (3-5), fallback с отчётом |
| Дорого по токенам | Высокая | Budget control: пользователь задаёт лимит |
| Ложноположительные тесты | Низкая | Только тесты, которые были до изменений |
| Не все проекты имеют тесты | Средняя | Fallback на build check + lint |
| Сложность для инфраструктурных изменений | Высокая | Размер изменения: если diff > 200 строк — отдать пользователю |

## 7. Резюме

**Проблема:** Compaction и Skills — крутые архитектурные фичи, но невидимые для нового пользователя.

**Решение:** Fix-Until-Green — visible killer feature, понятная с первых секунд: «агент не бросает код, пока он не работает».

**План:** В Phase 3 мы делаем TUI Polish (без него агент недружелюбен) + MVP Fix-Until-Green (киллер-фича) + CI/CD.

**Месседж:** *«SOBA — единственный CLI-агент, который доводит код до зелёного. Не пишет и забывает, а пишет, проверяет и чинит, пока не заработает.»*
