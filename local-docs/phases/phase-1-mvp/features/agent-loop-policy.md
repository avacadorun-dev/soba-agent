# Agent Loop Policy

## Цель

Agent loop должен продолжать полезную работу до фактического завершения задачи, явного блокера или отмены пользователем.
Количество model/tool шагов само по себе не является признаком проблемы.

## Решение об остановке

Loop завершается, когда выполняется одно из условий:

1. Модель вернула финальный ответ без незавершённого coding-flow.
2. Модель явно сообщила блокер, требующий решения пользователя.
3. Пользователь отменил выполнение.
4. Stall guard подтвердил отсутствие прогресса и recovery не помог.
5. Достигнут настроенный пользователем аварийный лимит итераций или времени.

По умолчанию аварийные лимиты итераций и времени отключены. Это позволяет длинным задачам естественно превышать 25
шагов. Безопасность обеспечивается stall detection и возможностью отмены через `Ctrl+C`.

## Stall Detection

Одна tool-итерация считается не дающей прогресса, если:

- повторяет тот же набор инструментов с теми же аргументами и получает те же результаты;
- все вызовы инструментов завершаются ошибками несколько итераций подряд.

После порога stall guard добавляет recovery-сообщение с требованием пересмотреть задачу и сменить стратегию. Если
несколько recovery-попыток не помогают, loop останавливается с явной ошибкой.

Recovery и остановка публикуются как `loop_guard` события и отображаются в TUI, поэтому policy не работает скрытно.

## Диагностика

Флаг `--debug` добавляет в JSONL-файл сессии sidecar-записи `type: "debug"`. Они фиксируют начало хода, каждую
итерацию, ответ провайдера, auto-continue, причину остановки и завершение хода. Debug-записи не входят в дерево
диалога и никогда не отправляются модели.

Решение о продолжении text-only ответа основано только на структурных сигналах:

- до первого tool call видимый text-only ответ принимается как прямой ответ;
- после tool calls `commentary` запускает recovery/verification flow;
- `final_answer` завершает ход, если после мутаций есть проверка;
- провайдеры без стабильной поддержки `phase` могут завершить ход control tool `finish`;
- reasoning-only/whitespace-only ответы не считаются пользовательским ответом.

После `edit/write` требование верификации применяется к промежуточным ответам; явный `final_answer` всегда завершает
ход. Повторяющиеся промежуточные ответы без tool call или `finish` ограничены guard-лимитом и завершаются явной
ошибкой. `finish` не исполняется как проектный инструмент: перед завершением completion gate проверяет:

- отсутствие непроверенных изменений;
- отсутствие активных ошибок либо их явное acknowledgement;
- наличие конкретных completion criteria;
- наличие успешных tool calls текущего хода и verification evidence после последнего изменения файлов.

Tool-ошибки хранятся как append-only audit history со статусом `active`, `resolved` или `acknowledged`. Успешный
повтор той же операции переводит ошибку в `resolved`; это отделяет восстановленные сбои от актуальных проблем.
Evidence принадлежит loop и не передаётся через угадываемые моделью внутренние `call_id`. Отклонённый `finish`
возвращается модели как tool result с конкретными причинами. Три последовательных отклонения завершают ход через
loop guard, чтобы дефект completion contract не превращался в бесконечный поток проверок.
Completion contract добавляется и к custom system prompt, потому что это часть транспортного протокола loop, а не
пользовательская эвристика.

Интерактивный TUI показывает `Agent is idle` только после `turn_end` или ошибки. Между model/tool итерациями статус
остаётся `Agent is working...`, чтобы внутренняя пауза не выглядела как тихая остановка.

## Настройки

| Поле config.json | Env | CLI | Default |
|---|---|---|---:|
| `maxAgentIterations` | `SOBA_MAX_AGENT_ITERATIONS` | `--max-agent-iterations` | `0` (unlimited) |
| `maxStalledIterations` | `SOBA_MAX_STALLED_ITERATIONS` | `--max-stalled-iterations` | `4` |
| `maxRunMinutes` | `SOBA_MAX_RUN_MINUTES` | `--max-run-minutes` | `0` (unlimited) |

## Основания

- OpenAI Agents SDK позволяет отключить `maxTurns`, передав `null`, и рассматривает лимит как аварийный guard.
- LangGraph отмечает, что сложные workflow естественно могут превышать дефолтный recursion limit.
- pi-agent делает основной упор на stop reasons, abort, retry и compaction, а не на маленький фиксированный предел.
