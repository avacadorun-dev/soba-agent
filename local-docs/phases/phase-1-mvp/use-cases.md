# Фаза 1 — MVP: Use Cases

## Акторы

| Актор | Описание |
|-------|----------|
| **Senior Engineer** | Разработчик, работающий с крупной кодовой базой. Использует soba для рефакторинга, генерации кода, отладки. |
| **Fullstack Developer** | Разработчик, пишущий бэкенд и фронтенд. Использует soba как CLI-ассистента. |
| **Система (soba)** | Сам агент. Принимает решения о компакции, управляет контекстом. |

---

## UC-1: Первичный запуск и конфигурация

**Приоритет:** P0 (критический)

**Сценарий:**
1. Разработчик устанавливает soba глобально (`npm install -g soba-agent` или `bun add -g soba-agent`)
2. При первом запуске `soba` система проверяет наличие `~/.soba/config.json`
3. Если конфиг отсутствует, soba предлагает интерактивный first-time setup:
   - Ввод OpenAI-совместимого URL (по умолчанию `https://api.openai.com/v1`)
   - Ввод API-ключа
   - Выбор модели по умолчанию
   - Установка рабочей директории или автодетект
4. Конфиг сохраняется в `~/.soba/config.json`

**Ожидаемый результат:** После setup команда `soba "hello"` отправляет запрос к LLM и выводит ответ.

**Альтернативный поток:**
- `soba --api-key sk-... --model gpt-4o` — явное указание параметров без конфига
- `soba --base-url http://localhost:11434/v1` — подключение к локальной модели

## UC-2: One-shot выполнение задачи

**Приоритет:** P0

**Сценарий:**
1. Разработчик выполняет: `soba "Добавь валидацию email в src/auth.ts"`
2. soba загружает конфигурацию и создаёт in-memory сессию
3. Система отправляет prompt модели с системным промптом и инструментами
4. Модель вызывает инструменты: `read("src/auth.ts")`, затем `edit(...)`
5. soba исполняет инструменты, возвращает результат модели
6. Цикл повторяется до финального ответа
7. Результат выводится в консоль, сессия сохраняется (если не `--no-session`)

**Ожидаемый результат:** Файл `src/auth.ts` изменён согласно задаче. В консоль выведен результат.

**Граничные случаи:**
- Файл не существует → soba сообщает об ошибке, модель предлагает создать
- API-ключ невалиден → сообщение об ошибке и инструкция по настройке
- Модель возвращает ошибку (rate limit) → retry с экспоненциальной задержкой

## UC-3: Интерактивный режим (REPL)

**Приоритет:** P1

**Сценарий:**
1. Разработчик запускает `soba -i`
2. Система отображает приглашение и информацию о модели/сессии
3. Разработчик вводит запрос: `Отрефактори src/utils.ts — разбей на модули`
4. soba выполняет цикл (как в UC-2)
5. После ответа приглашение возвращается
6. Разработчик может задавать уточняющие вопросы, продолжая сессию
7. `↑/↓` позволяют вернуться к предыдущим запросам
8. При вводе `/` показывается список команд, а при вводе `@` — список файлов проекта
9. Выделенный мышью текст можно скопировать через `Cmd+C` или `Ctrl+Shift+C`

**Ожидаемый результат:** Полноценный диалог с агентом. Контекст сохраняется между запросами.

**Специальные команды:**
- `/compact [instructions]` — ручная компакция контекста
- `/rewind [checkpoint-id]` — перемотка к checkpoint
- `/session` — статистика сессии: effective context tokens и historical tokens
- `/budget` — использование effective context tokens
- `/model <name>` — смена модели
- `/exit` — выход

## UC-4: Продолжение предыдущей сессии

**Приоритет:** P1

**Сценарий:**
1. Разработчик завершил работу вчера (сессия сохранена)
2. Сегодня запускает `soba -c` (continue)
3. Система находит последнюю сессию для текущей директории
4. Загружает историю, восстанавливает контекст (с учётом compaction)
5. Разработчик продолжает работу: `Продолжи рефакторинг — что осталось?`
6. Модель на основе контекста понимает состояние и продолжает

**Ожидаемый результат:** Бесшовное продолжение работы, как будто сессия не прерывалась.

**Альтернативный поток:**
- `soba -s abc123` — возобновление конкретной сессии по ID
- `soba -r` — интерактивный выбор сессии из списка

## UC-5: Ручная компакция контекста

**Приоритет:** P0 (ключевая гипотеза)

**Сценарий:**
1. Разработчик работает над задачей уже 2+ часа
2. Контекст разросся, ответы модели замедлились
3. Разработчик вводит `/compact`
4. Система:
   - Определяет cut point (оставляет последние ~20K токенов)
   - Вызывает `POST /v1/responses/compact` (OpenResponses compact API)
   - Сохраняет CompactionEntry в сессию
   - Перестраивает input: [compactionItem] + [items после cut point]
5. Агент продолжает работу с чистым контекстом

**Ожидаемый результат:** 
- Контекст сокращён на 60-80%
- `/session` показывает уменьшившийся effective context; historical tokens сохраняются как audit-метрика
- `/budget` использует effective context, а не полный объём исторической JSONL-сессии
- Модель сохраняет понимание проделанной работы и контекст задачи
- Время ответа модели сокращается

**Граничные случаи:**
- Контекст менее 20K токенов → сообщение «контекст не требует компакции»
- OpenResponses compact API возвращает ошибку → retry

## UC-6: Rewind к checkpoint

**Приоритет:** P1

**Сценарий:**
1. После нескольких компакций разработчик понимает, что текущий путь неверный
2. Вводит `/rewind` — видит список checkpoint'ов
3. Выбирает checkpoint #2 («После рефакторинга auth»)
4. Система создаёт новую ветку от checkpoint #2
5. Контекст восстанавливается: summary checkpoint + сообщения после него
6. Разработчик продолжает с альтернативным подходом

**Ожидаемый результат:** Разработчик может экспериментировать с подходами без потери истории.

**Альтернативный поток:**
- `/rewind abc123` — немедленная перемотка к конкретному checkpoint по ID
- Несколько rewinds подряд — создаётся дерево альтернативных веток

## UC-7: Система доверенных доменов

**Приоритет:** P2

**Сценарий:**
1. Модель вызывает `bash("rm -rf node_modules")` — команда классифицируется как `dangerous`
2. soba запрашивает подтверждение: `Разрешить выполнение "rm -rf node_modules"? [y/N]`
3. Разработчик подтверждает или отклоняет
4. `read("src/auth.ts")` — классифицируется как `safe`, выполняется автоматически
5. `edit("src/auth.ts", ...)` — `normal`, выполняется с уведомлением

**Ожидаемый результат:** Безопасное выполнение операций без постоянных запросов подтверждения.

## UC-8: Отслеживание токен-бюджета

**Приоритет:** P2

**Сценарий:**
1. Разработчик запускает: `soba --budget 100000`
2. В начале сессии soba выводит: `Бюджет: 100,000 токенов`
3. После каждого ответа модели обновляется счётчик: `Использовано: 42,500 / 100,000 (42.5%)`
4. При достижении 80% бюджета: предупреждение
5. При достижении 100%: предложение увеличить бюджет или завершить

**Ожидаемый результат:** Прозрачный контроль затрат на токены.

## UC-10: Очередь сообщений во время работы агента

**Приоритет:** P1

**Сценарий:**
1. Агент выполняет длительную задачу
2. Разработчик отправляет дополнительные уточнения
3. Сообщения попадают в FIFO-очередь и видны в sidebar
4. Разработчик редактирует или отменяет отдельные queued-сообщения
5. После завершения текущего turn агент автоматически начинает следующее сообщение

**Ожидаемый результат:** Ввод во время работы агента не теряется и остаётся управляемым.

---

## UC-11: Scoped permissions

**Приоритет:** P1

**Сценарий:**
1. Опасная операция запрашивает разрешение
2. Разработчик выбирает одноразовое разрешение, разрешение конкретной операции на сессию или полный доступ к repo
3. Repo-scoped операции больше не спрашивают подтверждение
4. Сеть, privilege escalation, абсолютные пути и выход за пределы repo продолжают требовать подтверждения

**Ожидаемый результат:** Меньше повторных approval prompts без превращения repo access в неограниченный shell.

---

## UC-12: Прямые shell-команды из TUI

**Приоритет:** P1

**Сценарий:**
1. Разработчик вводит `!pwd` и видит stdout/stderr команды в ленте
2. Разработчик вводит `!!bun test`; команда выполняется, но stdout/stderr скрыты
3. Команды не отправляются модели и не требуют approval prompt
4. Во время активного turn команды попадают в FIFO-очередь с сохранением режима `!` или `!!`
5. `Ctrl+C` останавливает активную прямую shell-команду

**Ожидаемый результат:** Быстрые локальные команды выполняются прямо из TUI без участия модели.

## UC-9: Мультиязычный интерфейс

**Приоритет:** P1

**Сценарий (автоопределение):**
1. Разработчик с русской локалью (`LANG=ru_RU.UTF-8`) запускает `soba -i`
2. soba определяет язык через `LANG` → `ru`
3. Все сообщения CLI выводятся на русском:
   - Приглашение: `soba> `
   - Помощь: `Использование: soba [опции] [запрос]`
   - Ошибки: `Ошибка: API-ключ не найден`
   - Компакция: `Контекст сжат: 50000 → 12000 токенов`
4. OpenTUI chrome, slash-команды, setup wizard и print-renderer используют выбранный язык
5. Системный промпт, сообщения LLM, пути, команды и внешний текст ошибок не переводятся

**Сценарий (явное переключение):**
1. Разработчик запускает `soba -i --lang zh`
2. Все сообщения интерфейса — на китайском
3. В процессе работы переключает на английский: `/lang en`
4. Header, sidebar, status bar, placeholder и новый интерфейсный вывод немедленно переключаются без перезапуска
5. Уже выведенные сообщения остаются на исходном языке

**Сценарий (fallback):**
1. Неизвестная локаль (`LANG=fr_FR.UTF-8`)
2. soba не находит `fr` → fallback на `en`
3. Выводит предупреждение: `Language "fr" not supported, falling back to English`

**Ожидаемый результат:**
- Интерфейс на трёх языках без перезапуска
- Отсутствующие переводы прозрачно заменяются английским
- Все ключи и `{placeholder}` синхронны между тремя locale-файлами
- Однострочные элементы не ломают верстку при длинных переводах и узкой консоли

**Граничные случаи:**
- Перевод содержит `{var}` которого нет в вызове → оставить `{var}` как есть
- JSON-файл повреждён → fallback на `en` с предупреждением
- Ключ отсутствует во всех языках → показать сам ключ (не падать)
